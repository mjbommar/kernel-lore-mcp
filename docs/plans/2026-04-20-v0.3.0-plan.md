## Release strategy update (2026-04-21)

The original draft assumed a small `0.2.3` hardening release could
land ahead of the broader `v0.3.0` work. The live full-corpus
hosted-readiness run on `server6` changed that sequencing. We are
**not** cutting a standalone `0.2.3`; the would-be patch-line
hardening work is now the first code-bearing block of `v0.3.0`.

The concrete findings from that run:

- `kernel-lore-sync` + `kernel-lore-doctor --heal` successfully
  recovered 16 poisoned shard repos, but sync still needs to heal
  that state automatically instead of relying on a manual doctor
  step.
- `lore_regex` is **not public-ready on full corpus**. Even simple,
  list-scoped regex queries on `server6` consumed the full 5-second
  wall-clock budget and returned `query_timeout`.
- The cost-class caps are working, but `/metrics` did not surface
  client-observed `rate_limited` outcomes. We need end-to-end
  request metrics, not just tool-runtime histograms.
- Client-observed latency inflated sharply under concurrency even
  when server-side tool runtimes stayed low. That is a visibility
  gap first, a cap-tuning problem second.
- `_cached_corpus_stats` improved from timeout to ~2.2 s once warm,
  but still needs a generation-bound cache / warm path to be safe
  on a public box.
- `paths/vocab.txt` was still absent on the healthy full-corpus
  hosted box; `lore_path_mentions` remains a setup-quality paper
  cut until sync auto-builds it.
- Default hosted logs are too noisy. Tantivy merge / GC INFO spam
  hides the actual operator signals.

## Goal

Three things: *finish what we started* (close the v0.2.0 loose
ends so v0.3.0 is a clean surface), *make the hosted HTTP surface
safe enough for anonymous public traffic on a full-corpus box*,
and *make it easy for a first-time user to stand up a healthy
instance in one command*. No new research bets in this release —
the trained retriever and the snapshot-bundle reciprocity work
stay deferred to v0.4.x.

Concretely: cut v0.2.0, land the hosted-readiness hardening that
would otherwise have been `0.2.3`, remove the legacy grokmirror
scripts, ship a bootstrap flow, close the remaining perf cliffs,
and wire the handful of tools that still lag behind the updated
primitives layer.

## Principle: every item ends with a measurement or a structural invariant

No "we added a function." Every item in this plan has either a
latency target, a row-count target, or a CI-enforced invariant at
the end. If I can't write one, the item isn't well-scoped and
should be scoped down further before it lands.

## Phase 1 — close out v0.2.0 (blocking)

Must land before anything else in v0.3.0 ships. Small, mechanical.

### 1.1 Tag v0.2.0

- Rename the `[Unreleased]` section in `CHANGELOG.md` to
  `[0.2.0] — 2026-04-20`. Start a fresh `[Unreleased]` header.
- Tag the current `main` HEAD (`e7c6a60` at time of writing) as
  `v0.2.0`; push the tag.
- Update `Cargo.toml` + `pyproject.toml` version to `0.3.0-dev0` so
  nightly wheels from `main` don't shadow the tagged release.
- **Done when:** `git tag -l v0.2.0` returns the tag and the new
  [Unreleased] section is empty.

### 1.2 Remove the legacy grokmirror surface

Per the v0.2.0-beta commit (`aa04870`), the legacy chain is marked
LEGACY but still ships. v0.3.0 deletes it.

- Delete `scripts/klmcp-grok-pull.sh`, `scripts/post-pull-hook.sh`,
  `scripts/grokmirror*.conf`,
  `scripts/systemd/klmcp-grokmirror.{service,timer}`,
  `scripts/systemd/klmcp-ingest.{path,service}`.
- Delete the `KLMCP_GROKMIRROR_INTERVAL_SECONDS` and
  `KLMCP_INGEST_DEBOUNCE_SECONDS` fields from `Settings` (they've
  been deprecated since v0.2.0; grep no other call sites).
- Drop the runbook §0Z "legacy grokmirror path" section; update
  CLAUDE.md to remove the legacy-path mention.
- **Done when:** `grep -rni grokmirror src/ scripts/ docs/runbook` returns
  only historical references in docs/plans/.

### 1.3 Backfill side-table dates on release-clone data

The `backfill_side_table_dates` helper (#64) needs to run once on
every pre-v0.2.0 over.db. Bake it into `kernel-lore-sync`'s first-
ever run on a data_dir so users never hit the un-upgraded slow
path.

- On sync startup, probe the "any non-NULL date on a side-table
  row" invariant (one cheap indexed `SELECT ... LIMIT 1`).
- If the probe fails AND both side tables have >0 rows, run the
  chunked backfill inline before the first query is served;
  emit one-line progress to stderr every 50k rows.
- **Done when:** a deploy with a pre-v0.2.0 over.db sees
  `eq signed_off_by <email>` hit the <5 ms path on the second
  query after startup.

## Phase 2 — hosted-readiness hardening (blocking for public launch)

This is the would-be `0.2.3` line, now pulled into `v0.3.0`.
Nothing in the public launch track ships before these land.

### 2.1 Sync self-heal for poisoned shard repos

The `server6` incident proved `kernel-lore-doctor --heal` can
repair the bad state, but a public deployment should not require an
operator to discover and run a second command after sync gets
poisoned.

- Teach `kernel-lore-sync` to treat unopenable shard repos and
  zero-ref shard repos as corrupted local state: delete them and
  reclone automatically before ingest.
- Keep `kernel-lore-doctor --heal` as the explicit maintenance
  surface for operators, but document it as the narrow force-clean
  tool, not the primary recovery path.
- **Done when:** a synthetic zero-ref shard repo is auto-recloned
  in CI, and the exact poisoned-shard state seen on `server6`
  recovers via `kernel-lore-sync` alone with no manual filesystem
  cleanup.

### 2.2 Public-hosted posture for `lore_regex`

The live hosted test showed the current full-corpus regex path is
not viable for anonymous public traffic. A simple scoped regex
spent 5 seconds then timed out; the overload case cleanly produced
`query_timeout` + `rate_limited`, but the underlying tool is still
not usable.

- Decide the `v0.3.0` posture explicitly instead of leaving the
  current behavior in place: either disable `lore_regex` in hosted
  mode, or add enough admission control / narrowing to reject
  unsupported shapes fast.
- If a hosted regex path remains enabled, it must have a bounded
  candidate budget before body/patch scanning starts. No default
  "burn 5 s then timeout" path survives this phase.
- **Done when:** the public/hosted profile either rejects unsafe
  regex requests in <50 ms with actionable narrowing guidance, or
  completes representative list-scoped regex calls within the
  budget. The current full-corpus timeout behavior is gone.

### 2.3 End-to-end overload metrics + queue visibility

The client-side stress run saw `rate_limited` responses and large
tail-latency inflation under concurrency, but `/metrics` only told
part of that story.

- Add request-scope HTTP/MCP duration histograms and queue-wait
  histograms alongside the existing per-tool runtime metrics.
- Record `rate_limited` and every other non-`ok` status in
  Prometheus regardless of whether the call reached the tool body.
- Add per-cost-class in-flight gauges so we can see "expensive is
  saturated" directly instead of inferring it from client traces.
- **Done when:** the 48-concurrent moderate-load test records
  `rate_limited` in `/metrics`, and Grafana can distinguish tool
  runtime from queueing / transport overhead.

### 2.4 `lore_corpus_stats` generation cache + warm path

`_cached_corpus_stats` timed out once in the live test and later
returned in ~2.2 s. That's better than a hard timeout, but still
too close to the hosted wall-clock cap for a "summary" tool.

- Cache corpus-stats by generation instead of per-process best
  effort. Bust the cache on generation bump; optionally warm it on
  server start or first request after a sync.
- Wire the cached path into `/metrics` so we can see whether a
  timeout was on the cold-build path or on the steady-state path.
- **Done when:** warm `lore_corpus_stats` stays under 500 ms p95 on
  the full corpus and does not hit `query_timeout` in the
  adversarial suite.

### 2.5 Hosted deployment profile + log shape

The public box needs a documented, repeatable runtime posture —
not just a pile of env vars remembered by the operator who happens
to launch it.

- Introduce a deployment-profile knob (`local` / `hosted`, or an
  equivalent env name) that bundles the public-safe defaults:
  stricter expensive-tool posture, quieter logs, and any hosted-only
  admission control.
- Drop `tantivy::*` INFO chatter from the hosted default log
  filter. Keep our own progress / warning / error lines visible.
- **Done when:** one documented env toggle enables the exact
  runtime shape we intend to put on the open internet, and the
  default hosted logs are readable by an operator following a live
  sync or query incident.

## Phase 3 — onboarding polish (the "first five minutes")

The April 2026 review called out "workable for an infra-comfortable
user, not genuinely easy." Phase 3 is about closing that gap.

### 3.1 `kernel-lore-sync --bootstrap`

One-shot wrapper around the first-run sequence so new users
don't have to memorize four separate commands.

- New `--bootstrap` flag: runs sync with a small default include
  pattern (`/lkml/*` or the first-subsystem-the-user-asks-for);
  after ingest, auto-invokes `rebuild_path_vocab` + optional
  BM25 rebuild guarded by `--with-bm25`; prints a final
  capabilities banner + the next command to try (`kernel-lore-mcp
  status`).
- When invoked on an empty data_dir with no `--include`, print
  the disk/time estimate table from the README and require
  `--yes` before mirroring the full corpus.
- **Done when:** `kernel-lore-sync --bootstrap --data-dir $NEW
  --include '/wireguard/*' --yes` goes from empty dir to queryable
  MCP in under 5 min wall-clock on a local NVMe, with
  `capabilities.path_vocab_ready == true` on exit.

### 3.2 Auto-build path vocab during sync

The onboarding review caught `lore_path_mentions` returning
silently-empty because nothing ever built `paths/vocab.txt`. The
setup_required error (#72) is the stopgap; the real fix is that
sync builds it on every successful ingest.

- At the end of sync's ingest phase (after tid rebuild, before the
  manifest-cache save), call `path_tier::rebuild_vocab_from_over`
  unconditionally. ~1 s on a corpus the size of our changes per
  tick; no reason to gate it. This is no longer just onboarding
  polish — the hosted `server6` run proved it is also a launch-
  quality requirement.
- Add a settings flag `KLMCP_SKIP_PATH_VOCAB=1` as an escape
  hatch for operators who deliberately don't want the path tier.
- **Done when:** fresh-ingest `--bootstrap` results in
  `capabilities.path_vocab_ready == true` without an extra manual
  command; `test_surface_manifest.py` gains a coverage check
  asserting `lore_path_mentions` works on an ingested fixture
  without a manual rebuild step.

### 3.3 Fast-dev fixture mode

The synthetic shard under `tests/python/fixtures/` is ~2 messages
— enough for tests, useless for a human's first exploration of
the tool surface.

- Add `kernel-lore-sync --fixture demo` that writes a synthetic
  data_dir containing ~200 canned messages across 3 lists, all
  tiers built, so a reviewer can `kernel-lore-mcp serve` against
  it in seconds without touching lore.
- Source the canned messages from a small committed tarball under
  `scripts/fixtures/demo-corpus.tar.zst` — immutable, versioned,
  small enough to carry in the repo.
- **Done when:** `kernel-lore-sync --fixture demo --data-dir $TMP`
  + `./scripts/agentic_smoke.sh` against that data_dir passes
  every check including the path-mentions probe, with no network
  access.

## Phase 4 — remaining performance cliffs

The v0.2.0 work closed the big obvious ones. v0.3.0 picks up the
long-tail items that the follow-ups doc queued.

### 4.1 Cross-list `patch_search` cold first-query

Warm is 360 ms (#66); cold still pays ~9 s because the OS page
cache is empty for all 530 trigram segment files on first use.
The boot-time warmup probe (#70) helps but a single probe only
touches one or two segments worth of postings — the rest fault
in on first real query.

Approach: read-ahead at warmup time. Touch the `trigrams.fst` +
`trigrams.postings` + `docs.tsv` file headers for every segment
without decoding them (just open + read first 4 KB). Keeps the
warmup cost bounded (~few hundred ms) while populating the full
page-cache residency.

- **Target:** first cross-list `patch_search` p50 under 2 s on a
  cold VM after boot.

### 4.2 CI performance gate

No automated regression test for the lore-scale targets. A
ten-line change could accidentally reintroduce the 68 s
`patch_search` without anyone noticing until it hit an agent.

- GH Actions job runs `scripts/bench/bench_concurrent_mixed.py`
  + new `bench_fast_paths.py` against a committed 100k-msg
  fixture on every PR that touches `src/`.
- Threshold each bench against a json file in
  `docs/bench-targets.json` — fails the job if any p50 regresses
  by more than 20 %.
- **Done when:** one PR that deliberately regresses
  `eq from_addr` to the old sequential-scan path fails CI with
  a legible "p50 regressed 53× (0.4 ms → 21 ms), budget was 5 ms"
  message.

### 4.3 Popular-needle `patch_search` cap refinement

Common needles like `EXPORT_SYMBOL` currently fail fast with
`too many candidates`. That's correct fail-loud, but the error
is not actionable — the caller can't tell how to narrow.

- When the cap fires, do a second indexed pass that counts how
  many segments hit vs. the whole set; include that in the error
  so the caller knows whether narrowing by list will help.
- For needles that are just common trigrams (e.g. `the`), return
  `invalid_argument` up front instead of waiting for the trigram
  walk to hit the cap. Probe the fst once per needle — if >50 %
  of segments carry every trigram, reject as too-broad.
- **Done when:** the cap-hit path emits a message shaped like
  `{hit_segments: 487, total: 530, try_list: 'lkml'}` instead of
  just "too many candidates."

## Phase 5 — remaining tool surface

Small items that didn't make v0.2.0.

### 5.1 Cross-list maintainer graph

`lore_maintainer_profile` currently answers "who is declared / who
actually touches this path." Agents asking deeper questions — "who
should I Cc on a networking patch" — need the flip side: for a
given person, every MAINTAINERS section they're listed in, every
path they've actually touched with counts and recency.

- New tool `lore_maintainer_graph(addr)` returns:
  - `declared_in: [{section: "NETWORKING DRIVERS", role: "R",
    paths: [...]}]` (from parsed MAINTAINERS).
  - `observed_paths: [{path, touches, last_unix_ns}]` (from
    `over_touched_file` JOIN `over_trailer_email` on
    signed_off_by).
  - `co_reviewers: [{addr, count, last_unix_ns}]` — everyone who
    appeared on a trailer alongside the target within the same
    message.
- Reuses #64 covering indexes; no new storage.
- **Target:** p50 <100 ms for a typical maintainer on the full
  corpus.

### 5.2 Thread-state classifier upgrade

`lore_thread_state` (#69) now has authoritative `merged` via git-
sidecar. The rest of the ladder (rfc / superseded / nacked /
under_review / abandoned / unknown) is still keyword-heuristic.

- Train a tiny logistic regression (or a single decision tree, to
  keep CPU inference free) on features we already compute: trailer
  counts, reply-graph depth, gap-since-last-activity, normalized-
  subject revision count, etc. Keep it to one Python file;
  inference must run in the Rust reader's budget.
- Promote `confidence` from the current high/medium/low enum to a
  calibrated float in `[0, 1]`, with the enum kept as a public-
  facing bucket.
- **Done when:** a 500-thread hand-labeled eval set lifts the
  overall accuracy vs. the current classifier by at least 10
  points and doesn't regress any of the existing buckets'
  precision.

### 5.3 Subject-normalized covering index

The F2a-residual subject_normalized column landed in v0.2.0 but
the index is a plain b-tree on a single column. Queries like
"every 'Re: [PATCH] ksmbd: fix oops'" over the full corpus still
do a sort — the same issue #64 fixed on the trailer side.

- Same shape: partial index on `(subject_normalized, date_unix_ns
  DESC)`. One-line schema change + the same CTE rewrite in
  `scan_eq`.
- **Target:** `eq subject_normalized <popular-subject>` p50 under
  5 ms on the full corpus.

## Phase 6 — ops maturity

### 6.1 HTTP transport end-to-end test

Current tests exercise the stdio path. The Streamable-HTTP
transport is also shipped but only smoke-tested.

- New `tests/python/test_http_transport_e2e.py` that stands up the
  full `mcp.http_app()` via `uvicorn`, runs an agent-shaped
  workflow (search → expand_citation → thread → patch), asserts
  cursor pagination round-trips, asserts `/status` +
  `/metrics` return sensible bodies concurrent with an in-flight
  query.
- **Done when:** the HTTP-transport test is part of CI and catches
  a hand-crafted regression (e.g. accidentally closing the
  connection mid-response) before merge.

### 6.2 Capability-gated metric series

The `/metrics` gauges currently report zero for unbuilt tiers as
if they were at generation 0. That's misleading on fresh deploys.

- Switch to a `klmcp_capability{name="over_db",ready="true"} 1`
  shape so Prometheus sees "absent" vs "present-but-zero" as
  distinct states.
- Add a `klmcp_side_table_backfill_remaining{table="..."} N`
  gauge so the one-time date backfill (phase 1.3) shows up in
  existing alerts.
- **Done when:** Grafana can render a "which deployments still
  need the pre-v0.2.0 backfill" panel without custom glue.

### 6.3 Systemd OnFailure= wiring

`klmcp-sync.service` exit codes are distinct (0 / 2 / 3) but the
shipped unit doesn't actually alert on any of them.

- Add `OnFailure=klmcp-alert@%n.service` template with a stub
  implementation that writes a single-line event to the journal
  using a dedicated syslog facility. Operators can override to
  page / Slack / whatever.
- Document the exit-code → alert-shape mapping in the runbook.
- **Done when:** a simulated manifest-fetch failure leaves a
  journal entry tagged with the alert facility without the
  operator having to post-process systemd output.

## Phase 7 — deferred research (explicitly NOT in v0.3.0)

Here so it doesn't get silently re-added under a feature flag.

- **Trained kernel retriever.** North star per
  `docs/research/training-retriever.md`. Multi-month; v0.4+.
- **Snapshot-bundle publication (reciprocity).** Depends on the
  hosted instance going live, which is its own track.
- **Patchwork integration.** The `lore://patchwork/{mid}` stub
  resource stays a stub until patchwork.kernel.org's API is
  integrated — that's a coupling with external infrastructure
  we're not committing to in a v0.3.0 timebox.
- **CVE-chain tool.** Depends on the security posture decisions
  in `docs/ops/threat-model.md`; not a code-level decision.
- **OAuth 2.1 / auth-gated tiers.** Same constraint as v0.1.x —
  anonymous-read-only is a product commitment, not a technical
  gap.
- **Per-IP cost-class fairness.** Known-gaps section of
  `docs/ops/production-hardening.md`. Only needed once we actually
  deploy a multi-tenant hosted instance.

## What I am NOT doing, deliberately

- No schema bumps. `over.db SCHEMA_VERSION` stays at 1. Every
  change in this plan is additive (new columns via ALTER, new
  side tables, new indexes).
- No new dependencies at the Rust layer. If an item requires a
  new crate, it gets justified in its own commit message, not
  buried in a feature.
- No new cost-class. The 3-tier (cheap / moderate / expensive)
  envelope in `cost_class.py` covers everything here.
- No "framework" layer. Each tool that wants pagination already
  uses the `cursor.py` helpers directly; no generic
  `PaginatedTool` base class even though it's tempting — three
  concrete implementations is cheaper than the abstraction and
  catches the review-reported drift problems earlier.
